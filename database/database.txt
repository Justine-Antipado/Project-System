Hindi mo kailangang lagyan ng **Delete (Del) button** ang bawat table sa UI mo. Dahil sa magandang pagkaka-design ng database schema mo (na may `ON DELETE CASCADE`), mas matalino at mas malinis sa UI kung sa mga **pangunahing tables (parent tables)** mo lang ilalagay ang delete function.

Kung ilalagay mo ang Delete button sa lahat, pwedeng maging magulo ang user experience (UX) at baka makabura ang admin ng mga importanteng data nang hindi sinasadya dahil sa domino effect.

Heto ang gabay kung aling mga tables ang **dapat** at **huwag** lagyan ng Delete button sa UI mo:

---

### 1. Mga Tables na DAPAT may Delete Button (Parent Tables)

Ito yung mga hawakan ng pangunahing impormasyon. Kapag binura mo ito, malaking chunk ng data ang mawawala, kaya lagyan mo ito ng **Confirmation Modal** (e.g., *"Sigurado ka ba? Mabubura rin ang lahat ng nakakabit na data..."*).

* **`students`** — Kailangan ito kapag may nag-drop, nag-transfer, o nag-graduate na estudyante.
> *Cascade Effect:* Kapag binura ang estudyante, awtomatikong mabubura ang records niya sa `officers` at ang mga attendance logs niya sa `event_attendance`.


* **`events`** — Para sa mga kanseladong events o maling entry.
> *Cascade Effect:* Mabubura lahat ng attendance logs ng mga estudyante na naka-scan sa event na iyon.


* **`organizations`** — Sakaling mag-disband o matapos ang termino ng isang partikular na grupo.
> *Cascade Effect:* Awtomatikong matatanggal ang mga officers sa ilalim ng org na iyon.


* **`school_years`** — Bagamat madalang gamitin ang delete dito (dahil sa history ng school), kailangan pa rin ito kung sakaling nagkamali sa pag-input ng taon (e.g., Typo mula "2026-2027" naging "2026-2022").

---

### 2. Mga Tables na HINDI KAILANGAN o MAAARING TANGGALIN ang Delete Button

* **`semesters` (Huwag na o Itago)**
* *Bakit?* Ang semester ay karaniwang static (1st Semester, 2nd Semester, Summer). Mas magandang i-manage ito sa loob mismo ng view o form ng School Year. Kung buburahin mo ang isang semester, madadamay ang lahat ng events sa semester na iyon. Mas ligtas na walang direktang "Del" button ito sa isang hiwalay na listahan.


* **`event_attendance` (Huwag lagyan ng generic Delete)**
* *Bakit?* Ito ay isang transaction table (mga logs ng scan). Hindi ka dapat nagbubura ng attendance basta-basta para sa integridad ng system.
* *Alternatibo:* Kung nagkamali ng scan (halimbawa, maling estudyante ang nai-scan), gumawa ng **"Remove from Event"** o **"Void Scan"** button sa loob ng event summary view, sa halip na isang hilaw na Delete button.


* **`officers` (Gawin mong "Revoke/Remove Position" sa halip na Delete)**
* *Bakit?* Ang officer ay nakatali sa isang estudyante. Kung gusto mong alisin ang pagiging officer niya, mas magandang UI action ang tawaging **"Remove as Officer"** sa profile ng student o sa listahan ng org, sa halip na i-trato ito bilang pagbura ng isang independent row.



---

### Ang Rekomendasyon sa UI Flow Mo

Imbes na bigyan ng kapangyarihan ang user na mag-delete kahit saan, gamitin mo ang ganitong diskarte sa UI para maging malinis at ligtas ang system mo:

1. **Gumamit ng Tabbed o Dashboard View:** UI Layout.
Igrupo mo ang `school_years` at `semesters` sa iisang "Settings" o "Academic Year" page. Ang Delete button ay nasa tabi lang ng buong School Year.


2. **Magpatupad ng Soft Delete o Confirmation:** Safety Alert.
Para sa `students` at `events`, kapag pinindot ang Delete, magpapakita ang isang modal: *"Mabubura rin ang XX attendance records nito. Ituloy?"* bago mag-fire ang `DELETE` query.


3. **Gawing 'Action-Based' ang nasa Attendance at Officers:** UX Detail.
Sa Attendance List ng isang event, palitan ang basurahan (Delete) icon ng text o icon na **"Void Scan"** para mas swak sa konteksto ng ginagawa ng user.

Heto ang direktang gabay kung aling mga tables lang ang **DAPAT at HINDI DAPAT** lagyan ng Search Bar at Add Form batay sa logic ng project mo:

---

### 1. Mga Tables na DAPAT may Search Bar at Add Form (Independent Views)

Ito yung mga dynamic tables na araw-araw o madalas hahawakan ng admin. Mabilis lumaki ang data rito kaya mandatory ang Search Bar, at kailangan ng mabilisang pag-add.

* **`students`** — **DAPAT MAYROON (Pareho).**
* *Bakit?* Daan-daan o libo ang estudyante sa OMSC. Imposibleng mahanap ang isang student kung walang Search Bar (by Student ID o Last Name). Kailangan din ng Add Form para sa mga bagong enrollees o transferees.


* **`events`** — **DAPAT MAYROON (Pareho).**
* *Bakit?* Dito magsisimula ang attendance mo. Kailangan ng Add Form para sa mga bagong school activities, at Search Bar para mabilis mahanap ang nakaraang events.


* **`organizations`** — **DAPAT MAYROON (Pareho).**
* *Bakit?* Para sa madaling pag-add ng mga bagong accredited orgs at paghahanap sa kanila kapag marami na (e.g., JITS, CSC, atbp.).



---

### 2. Mga Tables na may "Kakaibang" Diskarte (Huwag lagyan ng generic Search/Add)

Dito papasok ang pagiging madiskarte mo bilang full-stack developer para maging malinis ang UX (User Experience) mo:

#### A. `school_years` at `semesters` (I-grupo sa iisang "Academic Settings" View)

* **Search Bar:** **HUWAG NA.** Ang School Year at Semester sa isang eskwelahan ay napakakonti lang (karaniwang less than 10 entries sa loob ng ilang taon). Hindi mo kailangan mag-search para sa "2026-2027" kung scrollable naman ito sa isang maikling listahan.
* **Add Form:** **PAGSAMAHIN NA LANG.** Sa halip na magkahiwalay na page, gumawa ka ng isang "Academic Year" page. Kapag nag-add ka ng School Year (e.g., 2026-2027), may checkbox o dropdown na doon para automatic malikha ang 1st at 2nd Semester sa ilalim nito.

#### B. `event_attendance` (Attendance Logs)

* **Search Bar:** **DAPAT MAYROON (by Student ID/Name).**
* *Bakit?* Kapag nagrereklamo ang isang estudyante na *"Hala, bakit absent ako sa Assembly?"*, kailangan ma-search ng admin ang pangalan niya sa listahan ng mga naka-scan para ma-verify.


* **Add Form:** **HUWAG GAMITAN NG TRADITIONAL FORM.**
* *Bakit?* Ang "Add" action dito ay dapat nangyayari sa pamamagitan ng **QR Code Scanner Module** (kapag na-scan ang QR, automatic pumapasok sa table).
* *Alternatibo:* Maglagay lang ng isang simpleng *"Manual Input"* button para sa mga estudyanteng nasiraan ng phone o walang QR code, pero sa loob lang ito ng mismong Event View.



#### C. `officers` (Organization Officers)

* **Search Bar:** **MAAARING MERON O WALA (Depende sa Layout).** Kung ang officers list mo ay nakapaloob na sa view ng isang partikular na Organization (e.g., Pag-click sa JITS Org, lalabas ang officers nito), hindi na kailangan ng hiwalay na search bar dahil maikli lang ang listahan ng officers kada org.
* **Add Form:** **GAWING "ASSIGN" BUTTON.** Huwag gumawa ng form kung saan ita-type mo uli ang pangalan ng officer. Gumawa ka ng button na **"Assign Officer"** kung saan pipiliin mo lang ang estudyante mula sa umiiral na `students` table at pipiliin ang posisyon niya (President, VP, etc.).
